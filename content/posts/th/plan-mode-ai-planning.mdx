## "ทำเลย ไม่ต้องคิดเยอะ" — ผิดแล้ว

![Plan Mode — ให้ AI วางแผนก่อนทำ](/images/blog/plan-mode.jpg)

ผมเคยใช้ Claude Code แบบนี้:

สั่ง → รอ → ดูผลลัพธ์ → ไม่ตรงกับที่คิด → แก้ → สั่งใหม่ → ยังไม่ตรง → แก้อีกรอบ

บางงานแก้ 3-4 รอบกว่าจะได้สิ่งที่ต้องการ

ไม่ใช่เพราะ Claude ไม่ฉลาด — แต่เพราะ **ผมไม่ได้ให้มันวางแผนก่อน**

แล้ววันหนึ่งผมค้นพบ Plan Mode — และตั้งแต่นั้นมา workflow ผมเปลี่ยนไปเลย

---

## Plan Mode คืออะไร?

Plan Mode คือ mode ที่บอกให้ Claude **หยุดลงมือทำทันที** — แต่ให้:

1. **อ่านและ explore** บริบทของงานก่อน (ไฟล์, โครงสร้าง, context)
2. **คิดออกแบบ approach** ว่าจะทำยังไง
3. **Present แผน** ให้คุณดูก่อน
4. **รอ approve** แล้วค่อยลงมือจริง

เหมือนมีพนักงานที่ฉลาดพอที่จะมาถามว่า "ผมจะทำแบบนี้ โอเคไหมครับ?" ก่อนที่จะลงมือทำงาน 3 ชั่วโมงแล้วได้ผลลัพธ์ผิดทั้งหมด

<Callout variant="info" title="ทำไมต้อง Plan ก่อน?">
งานที่ซับซ้อน Claude ต้องตัดสินใจหลายอย่างระหว่างทาง ถ้าไม่รู้ว่าคุณต้องการอะไรจริงๆ มันจะเดาเอา — แล้วเดาถูกได้แค่บางครั้ง Plan Mode แก้ปัญหานี้ด้วยการให้คุณ align กันก่อน
</Callout>

## เปิด Plan Mode ยังไง?

วิธีง่ายที่สุด — พิมพ์ `/plan` ก่อนสั่งงาน:

```
/plan สร้าง carousel 7 slides สรุปผลแคมเปญเดือนมีนาคม
ข้อมูลอยู่ใน clients/foodstock/reports/march-2026.csv
```

Claude จะตอบกลับมาด้วยแผน ไม่ใช่ผลลัพธ์ทันที เช่น:

> "ผมจะทำแบบนี้ครับ:
> 1. อ่าน march-2026.csv ก่อน ดูว่ามี metrics อะไรบ้าง
> 2. Slide 1: Cover — ชื่อแคมเปญ + เดือน
> 3. Slide 2: Overview — KPI หลัก 3 ตัว
> 4. Slides 3-5: Deep dive แต่ละ channel
> 5. Slide 6: Top performing content
> 6. Slide 7: Insights + Next steps
> Format: 1080x1080, style เดียวกับ template ที่มีอยู่
>
> โอเคไหมครับ? หรือต้องการปรับลำดับ/เพิ่ม slides ก่อน?"

ตอนนี้คุณมีโอกาส **แก้ก่อนลงมือ** — ถูกกว่าแก้หลังทำเสร็จมาก

---

## ตัวเลขจริงจาก Agency

ผมใช้ Plan Mode ทุกวันในงาน agency และวัดผลได้ชัด:

| Metric | ก่อน Plan Mode | หลัง Plan Mode |
|--------|--------------|---------------|
| **รอบการแก้งานเฉลี่ย** | 3-4 ครั้ง | 1 ครั้ง |
| **เวลาที่ประหยัดได้** | — | ~40% |
| **งานที่ออกมา "ตรง" ครั้งแรก** | ~30% | ~80% |

40% ไม่ใช่ตัวเลขที่ประเมินเอง — ผมวัดจากงาน carousel จริง 12 ชิ้นในเดือนกุมภาพันธ์ กับมีนาคมที่เริ่มใช้ Plan Mode เต็มตัว เวลารวมต่อชิ้นลดลงเฉลี่ย 38 นาที

## ตัวอย่างจริง: Carousel 7 Slides

นี่คือตัวอย่างที่เกิดขึ้นจริงเมื่อสัปดาห์ที่แล้ว ผมต้องทำ carousel สรุปผล campaign ให้ลูกค้า food hall

### ไม่ใช้ Plan Mode (แบบเดิม)

```
ผม: สร้าง carousel 7 slides สรุปแคมเปญ LINE OA เดือนนี้
```

Claude เริ่มทำทันที — ผลิต slides ออกมาใช้เวลา 8 นาที

ผลลัพธ์: ดีในแง่ visual แต่ลำดับ slides ผิด ใส่ตัวเลขผิด metrics ไม่ตรงกับที่ลูกค้าสนใจ

แก้รอบ 1: ขยับ slides, เพิ่ม CTR เข้าไป
แก้รอบ 2: เปลี่ยน tone เป็น positive มากขึ้น
แก้รอบ 3: เพิ่ม benchmark เปรียบเทียบเดือนก่อน

รวมใช้เวลา **42 นาที**

### ใช้ Plan Mode

```
/plan สร้าง carousel 7 slides สรุปแคมเปญ LINE OA
ลูกค้าสนใจ: open rate, new member, coupon redemption
ต้องการ tone: positive แต่ honest, มี benchmark เดือนก่อน
ไฟล์ข้อมูล: clients/foodstock/line-oa-march.csv
```

Claude ตอบกลับด้วยแผน slide-by-slide ผมดู approve แล้วบอก "เพิ่ม CTR ไว้ใน slide 4 ด้วยนะ" — แค่นั้น

Claude ลงมือทำ ผลลัพธ์ออกมาตรง แก้แค่รอบเดียว (เปลี่ยน font ให้ใหญ่กว่านี้)

รวมใช้เวลา **22 นาที**

## เมื่อไหร่ควรใช้ Plan Mode?

Plan Mode ไม่ต้องใช้กับทุกงาน — ใช้เมื่องานมีลักษณะเหล่านี้:

| ใช้ Plan Mode | ไม่จำเป็นต้องใช้ |
|--------------|----------------|
| งานที่มี output หลายชิ้น (carousel, report, series) | แปลประโยคเดียว |
| งานที่ต้องการโครงสร้างเฉพาะ | สรุป text ที่ให้มา |
| งานที่มี files หลายไฟล์ประกอบ | ถามคำถามตรงๆ |
| งานที่ทำครั้งแรก (ยังไม่เคยทำแบบนี้) | งาน routine ที่ทำซ้ำๆ |
| งานที่ถ้าผิดแล้วแก้ยาก | ตอบเมล draft สั้นๆ |

สรุปง่ายๆ: **งาน complex หรืองานที่ output ต้องตรง spec → ใช้ Plan Mode**

## เทคนิคเพิ่มประสิทธิภาพ Plan Mode

### 1. ให้ข้อมูล context ตอน plan

```
/plan [งานที่ต้องการ]
context: [ข้อมูลที่ Claude ต้องรู้ก่อน]
constraint: [ข้อจำกัดหรือ spec ที่ต้องทำตาม]
output format: [อยากได้อะไร]
```

ยิ่ง context ชัด → แผนที่ได้ยิ่งตรง

### 2. Review แผนจริงๆ อย่า approve ปาดเดียว

หลายคนพอเห็นแผนก็ approve ทันทีโดยไม่ได้อ่าน — ทำแบบนี้ Plan Mode ก็ไม่ช่วยอะไร

ใช้เวลา 30 วินาทีอ่านแผน ดูว่า:
- ลำดับสมเหตุสมผลไหม?
- มี step ไหนที่ขาดไปไหม?
- ตรงกับสิ่งที่คุณต้องการจริงๆ ไหม?

### 3. แก้แผนก่อนอนุมัติ

ถ้าเห็นว่าแผนขาด step หนึ่ง บอกเลย:

```
ดีครับ แต่เพิ่ม step สำหรับ mobile preview ด้วย
แล้ว slide 3 ขยับมาเป็น slide 2 แทน
approve ครับ
```

Claude จะ update แผนแล้วลงมือทำตามที่แก้ไขแล้ว

## Plan Mode กับงาน Agency จริง

ผมใช้ Plan Mode กับงานเหล่านี้ทุกสัปดาห์:

**งาน Content:**
- Carousel 5-9 slides → `/plan` ก่อนทุกครั้ง
- Blog post MDX → `/plan` วาง outline ก่อน
- Video script 60-90 วิ → `/plan` วาง shot list ก่อน

**งาน Report:**
- Monthly report ลูกค้า → `/plan` วาง section structure
- Data analysis → `/plan` วาง approach ก่อน query

**งานเทคนิค:**
- แก้ codebase → `/plan` ก่อน Claude ดู files
- สร้าง component ใหม่ → `/plan` ออกแบบ interface ก่อน

Pattern เดียวกันหมด: **ให้ Claude คิดก่อน ลงมือทีหลัง**

---

## ทำไม Plan Mode ถึงเปลี่ยน Game

ปัญหาหลักของการทำงานกับ AI คือ **expectation mismatch** — คุณนึกภาพ A ไว้ แต่ AI ส่ง B มาให้ แล้วก็แก้วนซ้ำ

Plan Mode แก้ปัญหานี้ด้วยการเพิ่ม **checkpoint ก่อนลงมือ**:

```
สั่ง → วางแผน → [CHECKPOINT: คุณ review + approve] → ลงมือ → ผลลัพธ์
```

Checkpoint นี้เองที่ประหยัดเวลาได้ 40%

ถ้าคุณใช้ Claude Code อยู่แล้วและยังไม่ได้ใช้ Plan Mode — ลองพรุ่งนี้เลยครับ สั่งงานชิ้นต่อไปด้วย `/plan` แล้วดูว่าผลต่างขนาดไหน

---

## สรุป

**3 สิ่งที่ต้องจำ:**

1. `/plan` → Claude วางแผนก่อน ไม่ลงมือทันที
2. Review แผนจริงๆ แก้ก่อน approve — ถูกกว่าแก้หลังเสมอ
3. ใช้กับงาน complex หรืองานที่ต้อง spec ตรง — ไม่ต้องใช้กับทุกงาน

ผมลด rework จาก 3-4 รอบเหลือ 1 รอบ และประหยัดเวลาได้ ~40% ในเดือนที่ผ่านมา ไม่ใช่เพราะ prompt เก่งขึ้น — แต่เพราะหยุดรีบสั่งงาน แล้วให้เวลา Claude คิดก่อน
